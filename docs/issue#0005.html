设计决策

决策 ContainsBy/None 复用 Any；All 直接实现

模糊处： SPEC §4.3 并列列出 Any/All/None/ContainsBy，未规定它们之间的实现关系。

选择： Any 直接实现（核心）；ContainsBy = s.Any(pred)；None = !s.Any(pred)；All 直接实现（不复用 Any）。

理由： ContainsBy 语义等同 Any（存在即真），None 是 Any 的逻辑否定，复用各一行即可，DRY 且 godoc 标注等价关系、测试（TestContainsByNoneEquivalent）锁定。All 若复用需写成 !Any(func(v T) bool { return !pred(v) })，引入否定闭包且双重否定损害可读性，直接实现（遇首个不满足即 false）更清晰。这些都是终止操作（非中间热路径），复用的单层方法调用开销可忽略，与 #3 中 Dedupe 内联（中间操作热路径）的取舍不同。

决策 Find（按值，顶层）与 FindBy（按谓词，方法）的命名区分

模糊处： SPEC §4.3 有 FindBy（方法），§4.4 有 Find（顶层），两者都叫「find」。

选择： Find[T comparable](s, v, def) T 按值查找（顶层，D4）；FindBy(pred, def) T 按谓词查找（方法，T any）。

理由： 遵循 D4——Find 需 v == x 比较（T comparable）故必须顶层；FindBy 用谓词不约束 T 故可为方法。By 后缀统一表示「按谓词/键」，与 ContainsBy/DedupeBy/UniqBy 等命名一致，读者可一眼区分「按值」与「按谓词」两个族。

权衡

权衡 Contains/IndexOf/Find 不抽取共享 helper

备选 A（当前）： 三个顶层函数各自直接实现 range + ==v 比较，各自短路。

备选 B： 抽取内部 indexOf(s, v) (int, bool)，让 Contains 复用 indexOf(...) != -1、Find 复用按索引取值。

胜出理由： 选 A。三者返回类型不同（bool / int / T），抽 helper 返回元组再各自解包，引入分支与额外复杂度；而每个函数体仅 5-8 行、逻辑直观。在单包小库语境下，B 属过度抽象，A 的轻微重复换取的是每个入口的独立可读性。

开放问题

问题 FindBy 的 def 参数无法区分「找到的值恰为 def」与「未找到」

不确定点： FindBy(pred, def) T 在未找到时返回 def（SPEC §4.3 规定）。但若找到的元素本身恰好等于 def，调用方无法区分「找到了 def」与「没找到」。

需确认： 是否需要 (T, bool) 形式的变体（如 FindByOpt），让调用方能明确区分「命中」与「未命中」？SPEC 当前只定义 def 形式。多数场景 def 足够（如用 -1/nil 哨兵），但值域不含哨兵的场景有歧义。属可选增强，需用户确认是否纳入。
