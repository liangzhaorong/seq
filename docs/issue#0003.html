设计决策

决策 Dedupe 采用内联实现，而非复用 DedupeBy

模糊处： SPEC §4.2 注脚只规定 Dedupe 因 D4 退化为顶层泛型函数，未规定它内部是否应复用 DedupeBy。

选择： 初版写成 return s.DedupeBy(func(v T) T { return v })（复用），review 后改为在内联闭包中直接比较 v != prev。

理由： 复用会在热路径上引入一个恒等闭包（func(v T) T { return v }），该闭包被返回的 Seq 捕获、逃逸到堆，且每个元素多一次间接调用——与 SPEC §8.2「保持中间操作闭包短小、利于编译器内联」的性能哲学相悖。内联后零额外开销，且与同文件 FilterNot/Compact 刻意内联的取法一致。代价是与 DedupeBy 重复约 10 行 first+prev 逻辑（见 Tradeoffs）。

决策 FilterNot 直接实现，不复用 Filter

模糊处： SPEC §4.2 只列出 FilterNot(pred func(T) bool) Seq[T] 的签名，未规定实现方式。

选择： 直接在新闭包中写 !pred(v)，而非 s.Filter(func(v T) bool { return !pred(v) })。

理由： 直接实现省去每次迭代额外一层 pred 包装闭包，与 Filter 结构对称、可读性更好。测试中用「等价于取反 Filter」子测试锁定两者语义一致。

决策 Compact 用 var zero T 判零值

模糊处： SPEC §1.3 D6 以表达式 v == *new(T) 描述 Compact 语义。

选择： 实现用 var zero T; ... if v != zero，godoc 同时引用 v == *new(T) 以桥接 SPEC 文字。

理由： var zero T 与 *new(T) 对所有类型产生完全相同的零值，但前者是更地道的 Go 写法，避免 new 的取址/解址。这是等价表达，非语义偏离。

权衡

权衡 Dedupe：内联 vs 复用 DedupeBy

备选 A（复用）： s.DedupeBy(identity)。优点：DRY，去重逻辑只在 DedupeBy 一处。缺点：恒等闭包逃逸堆 + 每元素间接调用，热路径有可测开销。

备选 B（内联，已选）： Dedupe 体内手写 first+prev 比较。优点：零额外开销、与 FilterNot/Compact 一致、契合 §8.2。缺点：与 DedupeBy 重复约 10 行逻辑，未来去重逻辑变更需同步两处。

胜出理由： 这是一个明确追求性能的惰性迭代器库（SPEC §8 反复强调零中间切片、闭包内联），且 Dedupe 是高频相邻去重入口；性能/一致性收益压过 DRY 收益。两处逻辑相同且有共享的测试范式（TestDedupe / TestDedupeBy），重复风险可控。

权衡 DedupeBy：first+prev 双状态 vs *K 单指针

备选 A（first+prev，已选）： first bool + prev K。

备选 B： prevp *K（初始 nil，prevp == nil || *prevp != k）合并为单状态。优点：少一个状态变量。缺点：引入指针解引用，且 first 并非冗余——若去掉它仅用 var prev K，当首元素键恰为 K 的零值时会误判为与初值重复而丢弃。review 据此拒绝该简化。

胜出理由： first 是正确性必需（防止零值键误丢），当前写法清晰、无运行时代价，指针化未必更可读。
