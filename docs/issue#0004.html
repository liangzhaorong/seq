Issue #4 — 截取与选取（中间操作） — 2026-07-06

设计决策

决策 TakeRight/DropRight 用环形缓冲，而非滑动窗口 slice

模糊处： SPEC §4.2/§5.3 只规定「需缓冲至多 n 个元素」，未规定缓冲数据结构。

选择： 环形缓冲 ring := make([]T, n); ring[filled%n] = v，第二趟按 (start+i)%n 回放。

理由： 滑动窗口写法 buf = append(buf, v); buf = buf[1:] 有两个隐患——底层数组持有已滑出元素的引用（经典 slice 内存泄漏，元素无法 GC），且 buf[1:] 后 cap 递减、append 可能扩容到 2n。环形缓冲固定 n 内存、无泄漏、无扩容，是「需缓冲 n」的最优实现。

决策 Slice 单趟索引扫描内联，不复用 Drop+Take

模糊处： SPEC §4.2 列 Slice 为独立方法，未规定实现方式（功能上等价 Drop(start).Take(end-start)）。

选择： 单趟 idx 扫描，start <= idx < end 时产出，idx >= end 立即停止上游。

理由： 复用 Drop(start).Take(end-start) 会引入两层闭包的间接调用（Drop 的 yield → Take 的 yield）；单趟扫描只遍历上游一次、零额外闭包，契合 SPEC §8.2 闭包内联哲学。godoc 已注明这一等价关系以防维护者重复造轮子。

决策 Slice 负索引归一化为 0（Go 风格，非 Python 从末尾）

模糊处： SPEC §4.2 只说「负索引/越界裁剪，godoc 注明语义」，未规定具体语义。

选择： 负 start/end 归一化为 0；start >= end 产出空流；end 超过流长度则取到末尾。

理由： 采用 Go 切片惯例（负索引非法，归 0），而非 Python 的「负索引从末尾计数」。Go 生态用户不会期望 Slice(-1, end) 表示倒数第一；归 0 语义简单、可预测、易测试。

决策 DropRight 的 n<=0 在闭包内单独透传，不 return s

review 触发： review 建议把 n <= 0 分支简化为方法顶层 return s（直接透传上游）。

选择： 拒绝该简化，保持「闭包内 nil 守卫 + n<=0 的 for-range 透传」结构。

理由： return s 当 s == nil 时返回 nil Seq，调用方 range 一个 nil Seq 在 Go 1.27 会 panic（这是本库每个操作符都必须 if s == nil { return } 的根因）。nil 守卫必须在闭包内、且在 range 之前。return s 绕过守卫，引入 panic。当前结构与 Drop/Take 等所有操作的 nil-guard-in-closure 范式一致。

权衡

权衡 环形缓冲 filled%n vs 写指针 wi=(wi+1)%n

review 建议： 用写指针 wi = (wi+1)%n 替代 filled%n，消除超长/无限流上 filled 累加到 int 溢出（变负 → 负索引 panic）的理论风险。

选择： 保留 filled%n。

理由： filled 是 int（int64），溢出需 2^63 个元素——在到达该数量前，make([]T, n)（TakeRight/DropRight 的缓冲分配）或流本身的内存早以 OOM 终止。该理论风险在物理上不可能先于 OOM 触发，而 filled%n 用单变量即可同时表达「写入位置、总数、回放起点」，比 wi + total 双变量更简洁。

权衡 Partition 双闭包各自迭代上游（要求可重入）vs 物化共享

备选 A（当前）： 返回两条独立闭包，各自 range s 一次并按 pred 过滤。要求上游可重入。优点：完全惰性、零物化。缺点：对一次性上游（channel-backed、有状态生成器）会静默给出错误结果（先消费的流拿全数据，后消费的拿空）。

备选 B： 物化 s 到 []T，两条流共享切片。优点：支持一次性上游。缺点：O(n) 内存，破坏惰性，且与库「零中间切片」哲学相悖。

胜出理由： 选 A。Of/FromSlice/Range/Generate 等主流构造器天然可重入，覆盖绝大多数场景；godoc 明确警告「上游必须可重复消费」。一次性上游是少数边缘情况，用文档约束优于牺牲惰性。
