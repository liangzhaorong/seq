Implementation Notes

Issue #2 — 映射与扁平化（中间操作） — 2026-07-05

Design Decisions

Decision 所有操作符加 nil 接收者守卫（库级模式确立）

Ambiguity: SPEC §5.5 规定 nil 接收者等价空流、不 panic；但实测 Go 1.27rc1 下 for v := range nilSeq 会 panic（invalid memory address），因为 range-over-func 会调用该 nil func。

Choice: 每个操作符闭包顶部加 if s == nil { return }。

Rationale: 这是 SPEC 明确的契约，且用户代码（var s Seq[int]、f 返回 nil、含 nil 的 Seq[Seq[U]]）可达。1 行/操作符的样板换正确性，值得。

Decision FlatMap/Flatten 把 nil 子流视为空

Ambiguity: f 返回 nil Seq[U]、或 Flatten 的子流为 nil 时应如何处理（会 panic）。

Choice: FlatMap sub := f(v); if sub == nil { continue }；Flatten if sub == nil { continue }。

Rationale: nil Seq 与 Empty 语义一致（皆不产出），视为空对用户友好；Flatten 的 godoc 也承诺「nil 子流不出元素」，必须为真。

Decision Tap 在 yield 之前调用 f

Ambiguity: Tap 的 f 在 yield 前还是后调用。

Choice: f(v) 然后 yield(v)。

Rationale: 元素流经即观察，符合 RxJS tap / Rust inspect 惯例；下游拒绝（yield false）时停止，未到达的元素不调用 f。

Deviations

Deviation 无重大偏差

Spec: SPEC §4.2 + §5.1。

Implemented: 完全按 SPEC 实现 Map/FilterMap/FlatMap/Tap + 顶层 Flatten，惰性与提前终止模板（§5.1）逐条遵守。

Why: 本 issue 的核心是验证 Go 1.27 范型方法（Map[U any] 等）可编译运行——已通过 build + lint + test 验证。范型方法是本库区别于 lo 的核心创新点，确认成立。

Tradeoffs

Tradeoff nil 守卫的样板 vs 文档化「禁用 nil 接收者」

Alternatives: (a) 每操作符 if s == nil { return }；(b) 不守卫，文档说明 nil 接收者未定义行为。

Pros/Cons: (a) 满足 §5.5、防御现实 nil，代价是 ~30 操作符各 1 行；(b) 更简洁但违背 SPEC、留 panic 隐患。

Why chosen: SPEC 明确要求 nil 安全，选 (a)。若样板失控可在 #16 前提炼私有 helper。

Tradeoff Flatten 与 FlatMap 共享嵌套循环但不合并

Alternatives: Flatten 委托 FlatMap(identity)。

Why chosen: 二者形态不同——Flatten 是 Seq[Seq[U]] 的顶层函数，FlatMap 是 Seq[T] 上投影返回 Seq[U] 的方法（SPEC §4.2 vs §4.4 都列出）。合并会模糊契约边界，保持独立。

Open Questions

Question nil 守卫样板是否需提炼为 helper？

Assumption: #3~#7 的所有操作符（Filter/Slice/Find/Aggregate/Collect 等）都需 if s == nil { return }。

Verify: 当操作符总数增长到 ~30 时，评估是否引入私有 helper（如 each[T](s, yield)）去重；目前 1 行/操作符尚可接受。

Question range-over-nil 是否会在 Go 1.27 正式版改变？

Assumption: Go issue #68931 讨论「允许 nil 迭代器」，若采纳则 for range nilSeq 变 no-op，守卫可移除。

Verify: Go 1.27 正式版发布后确认；守卫即便冗余也无害（保留即可）。
