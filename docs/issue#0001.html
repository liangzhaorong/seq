Design Decisions

Decision 互转原语 Unbox/FromSeq 归入 seq.go，与类型同处

Ambiguity: SPEC §2.4 把 FromSeq 列在 constructors.go，但 §3.2 又把它与 Unbox 作为「零成本互转」对一起定义。

Choice: Seq.Unbox 与 FromSeq 放在 seq.go（类型定义旁）；constructors.go 只放值构造器 Of/FromSlice/Empty/Range/Repeat/Generate/Cycle/FromMap。

Rationale: 互转原语是类型契约的一部分，与类型定义同文件便于阅读；值构造器是「产生数据」的独立关注点，分文件清晰。

Decision 在 #1 即引入 Seq2.Unbox 与 FromSeq2

Ambiguity: #1 验收列出的构造器是 Of/FromSlice/FromSeq/Empty/Range/Repeat/Generate/Cycle/FromMap，未列 FromSeq2；SPEC §3.2 也只为 Seq 定义了互转。

Choice: 额外加入 Seq2.Unbox() 与 FromSeq2()（各一行零成本转换）。

Rationale: FromMap 已产出 Seq2，没有互转原语则它无法喂给标准库 maps.*；对称补齐互转比把「类型引入」与「基础互转」拆到两个 PR（#1 与 #9）更连贯。#9 的 Seq2 操作可直接复用。

Decision Cycle 空源在无限模式下首遍空即终止

Ambiguity: SPEC §5.6 称无限流须配短路终止器；但 Cycle(Empty(), -1) 源永远产不出元素，消费者的 break 永无机会触发 → 死循环。

Choice: 无限分支中用 yieldedAny 追踪：若一整遍未产出任何元素，立即 return（产出空流）。

Rationale: Cycle 一个被完全过滤掉的流是「退化但可达」的输入；与其 100% CPU 死循环，空源应当产出空流，保持短路保证。

Decision Number 约束不含 complex

Ambiguity: 无（SPEC §3.3 明确「无 complex」）。

Choice: Number = int/uint/float 各宽度并集，排除 complex64/128。

Rationale: Sum/Product/Mean 对复数语义含糊，与 SPEC 一致，无歧义。

Tradeoffs

Tradeoff Range int 溢出：仅文档说明，不加守卫

Alternatives: (a) 每次迭代加溢出检查 i > math.MaxInt - step；(b) 预计算长度再 for range count；(c) 仅 godoc 标注 fixed-width int 限制。

Pros/Cons: (a) 严格保持半开区间不变量，但引入 math 依赖与每次迭代额外比较，收益仅限 near-MaxInt 边界；(b) 长度计算本身会溢出，治标不治本；(c) 零成本、诚实，符合标准库惯例（Go 无原生 Range，固定宽度算术溢出是固有性质）。

Why chosen: 实测确认溢出仅在 stop 接近 MaxInt 且 step≥2 时发生（Range(0,MaxInt,1) 实际正确终止），属 review-it 判定的「不现实边界」；按惯例文档化，不为不可达情形过度工程。

Tradeoff Repeat/Generate 负长钳制用 max(0,n)

Alternatives: if n < 0 { n = 0 } 三行 vs n = max(0, n) 一行。

Why chosen: max（Go 1.21+ 内置）更简洁地道，且集中一处语义；go1.27 已稳定支持。

Tradeoff Of 与 FromSlice 循环体近乎相同仍各自独立

Alternatives: Of 委托 FromSlice(vs) 复用循环体。

Why chosen: 二者契约不同——Of 是字面量可变参数（每次新切片），FromSlice 不拷贝且文档警告调用者勿改切片；循环体两行，去重反而模糊契约边界。保持独立。
