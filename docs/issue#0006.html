设计决策

决策 Reduce 用首元素作累加器初值（first 标志位）

模糊处： SPEC §4.3 给出 Reduce(merge) (T, bool) 签名，未规定累加器初值来源。

选择： 以首个元素为 acc 初值，用 first 标志位区分「空流」与「首元素」。空流返回 (zero, false)；单元素返回 (elem, true)。

理由： Reduce 语义是「元素间 merge」（同 Scala/lo 的 reduce），首元素自然作初值。若用类型零值作 seed 再 merge 首元素，会引入语义偏差（例如乘法归约、字符串拼接会多出一个零值/空串）。需要种子或异类型累加的场景由 Fold[U](seed, f) 覆盖。

决策 Min/Max 用 cmp.Compare（而非 < / >）

review 触发： 初版用 v < best / v > best；review 指出 SPEC §5.2 明确「Min/Max 用 cmp.Compare」。

选择： 改为 cmp.Compare(v, best) < 0（Min）/ > 0（Max），并 import "cmp"。

理由： 对齐 SPEC §5.2 字面；cmp.Compare 对 NaN 一致处理（视为最小，与标准库 slices.Sort 一致），而 < 对 NaN 行为不确定（NaN 与任何值比较都 false，会跳过更新）。虽然本库典型用法不含 NaN，但用 cmp.Compare 让边界可预测且与 SPEC/标准库对齐。

决策 ReduceRight 缓冲整个流（O(n) 内存）

模糊处： SPEC 未规定 ReduceRight 的实现方式。

选择： 物化整个流到 []T，再从右向左 merge(buf[i], acc)。

理由： range-over-func 迭代器单向（左→右），无法惰性从右归约；要看到末尾必须先消费到末尾。缓冲是唯一方案，godoc 已标注 O(n) 内存（SPEC §5.3 缓冲类操作惯例）。

权衡

权衡 Min/Max 直接实现，不复用 MinBy/MaxBy

备选 A（当前）： Min/Max 各自直接用 cmp.Compare 单趟扫描。

备选 B： Min = s.MinBy(func(a,b T) bool { return cmp.Compare(a,b) < 0 })，一行复用。

胜出理由： 选 A。Min 是顶层（T Ordered），MinBy 是方法（T any + less），复用需构造 less 闭包。直接实现避免闭包间接调用，与 #3 中 Dedupe 内联（而非复用 DedupeBy）的热路径决策一致。MinBy/MaxBy 仍保留供「自定义 less / 非 Ordered 类型」场景。

权衡 Mean 内部 float64 累加，不复用 Sum

备选 A（当前）： Mean 内部 sum := 0.0; sum += float64(v) 逐元素累加。

备选 B： float64(Sum(s)) / float64(Count(s)) 复用 Sum。

胜出理由： 选 A。Sum 返回 T（如 int），对超大整数流先求 int 和会溢出，再转 float64 已丢精度；直接 float64 累加避免中间 int 溢出，精度更优。代价是每个 int→float64 转换（可忽略）。
